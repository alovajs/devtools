# Worma 大规模 OpenAPI 生成性能优化方案

> 适用范围：`packages/worma`，从 `src/generate.ts` 进入的全链路生成流程。
> 目标场景：单份 OpenAPI 文档包含 1k–10k+ 接口的常态化生成与增量构建。

---

## 二、整体优化策略（三阶段合并落地）

将「立即收益」「中等改造」「大改造」三个阶段合并为一份完整方案。所有改造控制在 `packages/worma/src` 内，对外 API（`generate.ts`、`Config`、插件协议）保持兼容。

```
┌──────────────────────────────────────────────────────────────────┐
│  阶段 A 立即收益 ── 缓存与就地优化                                  │
│   A1 prettier 记忆化 + 只在最终文件级格式化                         │
│   A2 handlebars 编译/partials 缓存                                 │
│   A3 normalizer 循环检测换 WeakSet                                 │
│   A4 tagedApis 用 Map 索引                                         │
│   A5 outputFiles 并发写盘（p-limit 32）                            │
├──────────────────────────────────────────────────────────────────┤
│  阶段 B 中等改造 ── 数据结构与增量化                                │
│   B1 cloneDeep 路径精简 + $ref 解引用结果共享缓存                   │
│   B2 isEqual 改为 hash 表增量比较，支持 tag 级增量渲染              │
│   B3 .alova-cache.json 按 tag 拆分 + 流式写                         │
│   B4 Promise.all 加并发上限 (默认 64，可配置)                       │
├──────────────────────────────────────────────────────────────────┤
│  阶段 C 大改造 ── 自适应 Worker 池 + 流式管线                       │
│   C1 schema→TS 转换 worker 池（按接口数自动伸缩）                   │
│   C2 解析→转换→渲染→写盘 流式管线，背压控制                        │
│   C3 OpenAPI 解析与 Swagger2→3 转换并行化                           │
└──────────────────────────────────────────────────────────────────┘
```

## 三、详细修改方案

### 阶段 A · 立即收益

#### A1. Prettier 记忆化与延迟格式化

- **位置**：`src/core/parser/templateParser/helper.ts`、`src/astLoader/generates/utils.ts`
- **思路**：
  1. 移除 `normalizeCode` 内对每个 schema 调用 `prettier.format` 的逻辑；改为生成已规整的 AST 字符串（按缩进栈拼接）。
  2. 仅在 `helper/template/index.ts` 的 `outputFiles` 之前，对**整个最终文件**做一次 prettier（可选，由 `config.prettier=true` 启用）。
  3. 若确需保留 schema 级格式化，则增加 `Map<string, string>` 记忆化：以 `(canonicalSchemaKey + deep)` 为 key 缓存 prettier 结果；同一 `$ref` 必同输出，命中后直接复用。
- **预期**：3 万+ 次 prettier 降至 ≤ 输出文件数（≤数百），**时间下降 60–80%**。

#### A2. Handlebars 编译与 partials 缓存

- **位置**：`src/helper/template/index.ts`
- **改造**：
  - 在模块作用域引入 `templateCache: Map<string, Handlebars.TemplateDelegate>`，key 为 `path + mtimeMs`，命中即跳过 `compile`。
  - 将 `registerPartials` 提升到 `generateFromTemplateDir` 入口处一次性调用，并把已注册集合放入 `Set<string>` 去重；同一进程内重复 generate 不再重扫盘。
  - 新增 `clearTemplateCache()` 内部 API 供测试与 watch 模式调用。

#### A3. Normalizer 循环检测换 WeakSet

- **位置**：`src/astLoader/normalize/normalizer.ts:34/65/67/73`
- **改造**：将 `JSON.stringify` 序列化探针替换为 `WeakSet<object>` 维护的 `visiting` 集合，进入节点 `add`、退出 `delete`，命中即返回 `$ref` 占位。
- **预期**：消除每节点 O(s) 序列化开销。

#### A4. tagedApis 改 Map 索引

- **位置**：`src/core/parser/templateParser/index.ts:155`
- **改造**：在 `parseApi` 阶段以 `Map<tagName, TagedApi>` 暂存，最后一次性 `Array.from(map.values())`。
- **预期**：O(n·t) → O(n+t)。

#### A5. 并发写盘

- **位置**：`src/helper/template/index.ts` 的 `outputFiles`
- **改造**：
  - 引入 `p-limit`（已是项目现有依赖之一，不增加成本；若没有则使用项目内自有 `concurrencyLimit`）。
  - 先批量 `mkdir -p`（去重路径集合），再 `Promise.all` 并发 `fs.writeFile`，并发度默认 32，可配置 `config.io.writeConcurrency`。

### 阶段 B · 中等改造

#### B1. cloneDeep 路径精简 + $ref 解引用缓存

- **位置**：`src/core/parser/templateParser/helper.ts`、`src/utils/openapi.ts`
- **改造**：
  1. `apiMethod2ApiDescriptor` / `apiDescriptor2apiMethod` / `mergeObject` 内，仅在**用户 `handleApi` 钩子存在**时进行浅拷贝顶层，深拷贝下放到钩子内部由用户负责。
  2. `removeAll$ref` 抽出 `RefResolver`：以 `$ref` 路径为 key 缓存「已解析、已展开」的 schema（freeze 后只读）。同一文档内同名 `$ref` 解析一次，全程复用。
  3. 数组与原子节点不参与 clone，直接传引用。
- **预期**：每接口克隆次数从 4–6 次降至 ≤1 次。

#### B2. 增量比较 + Tag 级增量渲染

- **位置**：`src/helper/config/GeneratorHelper.ts:238`、`src/core/parser/templateParser/index.ts`
- **改造**：
  1. 为每个 API 计算稳定 hash：`xxhash64(canonicalJSON(api))`，作为接口指纹。
  2. `oldApis` 缓存中同步保存 hash 表 `Map<apiKey, hash>`；新一轮生成只比较 hash，不再 `lodash.isEqual` 全量树。
  3. 触发渲染时仅对 hash 变化的 tag 重新调用 `generateFromTemplateDir` 中对应的 `{tag}` 模板；全局模板（如 index.ts、types.ts）仍按聚合后 apis 整体渲染。
  4. 新增 `config.incremental: boolean`（默认 `true`）开关，便于排障。

#### B3. `.alova-cache.json` 拆分与流式写

- **位置**：`src/functions/wormaJson.ts`
- **改造**：
  - 缓存目录改为 `.alova-cache/`：`index.json`（meta + 各 tag 文件清单 + 文件级 hash） + `tags/{tag}.json`（每 tag 的 API 列表）。
  - 写入采用 `fs.createWriteStream` + 手动序列化（避免巨串）；
  - 读取按 tag 懒加载，`isEqual` 替换为 hash 比对（与 B2 联动）。
- **兼容性**：保留对旧 `.alova-cache.json` 的一次性迁移读取。

#### B4. Promise.all 节流

- **位置**：`src/core/parser/templateParser/index.ts:80/91`
- **改造**：
  - 引入项目内 `concurrencyLimit(limit)` 工具，默认 `Math.min(64, max(8, cpus*4))`。
  - `transformApiMethods` 与 `transformApis` 全部经过该 limiter，避免一次性产生 n 个 promise 链。
- **预期**：内存峰值与 GC 抖动显著下降。

---

### 阶段 C · 自适应 Worker 池 + 流式管线

#### C1. Schema → TS 转换 Worker 池（核心）

> Schema 归一化、AST 解析、代码生成为纯 CPU、无外部副作用，是天然可并行任务。

- **新增模块**：`src/core/workerPool/`
  - `index.ts`：导出 `SchemaWorkerPool`。
  - `worker.ts`：worker 入口；接收 `{ schema, options }`，调用现有 `astLoader.transformSchema` + `astGenerate` + `normalizeCode`，返回 `{ code, deps }`。
  - `pool.ts`：池实现，封装 `node:worker_threads`。

- **自动伸缩策略**（按接口数量决定线程数）：

  ```ts
  function pickPoolSize(apiCount: number): number {
    const cpu = Math.max(1, os.cpus().length);
    if (apiCount <= 200)    return 0;          // 直接主线程
    if (apiCount <= 1000)   return Math.min(2, cpu);
    if (apiCount <= 3000)   return Math.min(4, cpu);
    if (apiCount <= 8000)   return Math.min(Math.ceil(cpu * 0.75), cpu);
    return Math.max(2, cpu - 1);               // ≥8000 用满核心，留 1 核给主线程
  }
  ```

  - 允许配置覆盖：`config.performance.workerPool = 'auto' | number | false`。
  - `apiCount === 0` 或 `pickPoolSize` 返回 0 时退化为主线程同步路径，零额外开销。
  - 池采用懒启动（首个任务到达再 `spawn`）+ 空闲 30s 自动回收。

- **任务粒度**：以**单接口**为粒度过粗（IPC 占比高），以**单 schema**为粒度过细。采用**批处理**：每批 `Math.ceil(apis.length / workerCount / 4)` 个接口的 schema 集合一次性发给 worker，平衡 IPC 与并行度。

- **数据传输**：仅传只读 JSON（结构化克隆），避免函数引用；用户 `handleApi` 钩子仍在主线程跑（在 worker 之前的 transform 阶段完成）。

- **接入点**：`src/core/parser/templateParser/index.ts` 的 `transformApis` 阶段，把 `parseParameters / parseRequestBody / parseResponse` 的 schema→TS 部分委派给池。`schemasMap` 改为主线程汇总（worker 返回引用名），保持去重。

- **预期收益**（10000 接口、8 核机器）：CPU 利用率从 ~12% 提升至 ~80%，schema 阶段耗时近线性下降到 1/(workers)。

#### C2. 解析 → 转换 → 渲染 → 写盘 流式管线

- **位置**：贯穿 `templateParser/index.ts` 与 `helper/template/index.ts`
- **改造**：
  - 用 `AsyncIterable<ApiMethod>` 串联各阶段，避免「全量数组 + 全量 Promise.all」。
  - 主管线：`getApiMethods()*` → `transform()*`（节流并发） → `tag 聚合`（按 tag 攒批） → `渲染` → `写盘`。
  - 背压：写盘阶段慢时，前序阶段自动暂停（基于 limiter 队列长度阈值）。
  - 峰值内存：从 O(n·s) 降到 O(并发数·s)。

#### C3. OpenAPI 解析与 Swagger2→3 转换并行化

- **位置**：`src/core/parser/openApiParser/helper.ts`、`src/utils/base.ts`
- **改造**：
  - `importFresh` → `fs.promises.readFile` + `JSON.parse`，避免污染 `require.cache` 与同步阻塞。
  - 巨型远程 JSON 用 `stream-json` 流式解析，与下游 transform 形成管线。
  - `swagger2openapi.convertObj` 移入 worker（与 C1 池共享或单独一个 worker），避免主线程冷启动卡顿。


### M3 完成详情

| 编号 | 优化项           | 改动文件                                              | 实现方式                                                                                                                                                                                                                                                                                                                                     |
| ---- | ---------------- | ----------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| B2   | 增量哈希比较     | `GeneratorHelper.ts` + `helper/template/index.ts`     | ① 用 `crypto.createHash('sha256')` 为每个 API 计算 16 字符稳定指纹 + 聚合哈希 + 分 tag 哈希（`computeApiHash` / `computePerTagHashes`）；② `GeneratorHelper.generate()` 中替换 `lodash.isEqual` O(n·s) 全量深比较为 O(n) 哈希比对；③ `generateFromTemplateDir` 新增可选 `changedTags` 参数，仅重渲变更 tag 的模板（全局模板始终渲染）        |

**关键设计决策**：

- 哈希用 Node 内置 `crypto.createHash('sha256')`，16 字符前缀，无额外依赖
- 增量渲染：首次全量生成，二次仅渲染变化 tag（`diffChangedTags` 比较分 tag 哈希）
- `schemaVersion: 2` 版本号防止格式不匹配自动全量重建
- `readAlovaCacheJson` / `writeAlovaCacheJson` 保留作为后向兼容 API

测试结果：全部通过，含 `alovaCache.spec.ts` 6/6（验证新旧格式读/写/合并/排序兼容性）。

### M4 完成详情

| 编号 | 优化项              | 改动文件                                                     | 实现方式                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ---- | ------------------- | ------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| C1   | Schema→TS Worker 池 | `core/workerPool/` (新增 3 文件) + `templateParser/index.ts` | ① 新建 `worker.ts`：独立线程运行 `schemaLoader.transform()` + `astLoader.transformTsStr()`，返回主/嵌套类型（key+result 对）；② 新建 `pool.ts`：懒启动池（首次任务到达 spawn）、Round-Robin 分发、30s 空闲自动回收、`pickPoolSize(apiCount)` 自适应（<200→0/1k→2/3k→4/8k→0.75×cpu/≥8k→cpu-1）；③ 新建 `index.ts` 导出；④ `parseApiMethods` 中新增 `collectSchemaTasks` 预收集阶段 → worker 池并行处理 → 填充 schemasMap → 后续 `transformApis` 命中缓存秒出 |

**自适应缩放策略**：

```
apiCount ≤ 200    →  0 (主线程，零开销)
apiCount ≤ 1000   →  2 workers
apiCount ≤ 3000   →  4 workers
apiCount ≤ 8000   →  0.75 × cpus
apiCount ≥ 8000   →  cpus - 1 (留 1 核给主线程)
```

**关键设计决策**：

- Zero-dependency: 全部使用 `node:worker_threads` 内置模块
- 数据传递：`workerData` 传递共享 document+config（一次），批量消息传递 schema 列表
- 优雅降级：`pickPoolSize` 返回 0 时完全跳过，零额外开销；worker 出错自动 reject
- Worker 内初始化 `globalThis.ALOVA_WORMA_CONFIG` 避免缺失全局状态

### M5 完成详情

| 编号 | 优化项   | 改动文件                                                        | 实现方式                                                                                                                                                                                                                                                                             |

| C2   | 流式管线 | `helper/template/index.ts` + `helper/config/GeneratorHelper.ts` | ① 新增 `streamGenerateFromTemplateDir` 方法：按 tag 逐批渲染→立即写盘→释放内存，峰值内存从 O(n·s) 降到 O(max_tag·s)；② 保留 `generateFromTemplateDir` 作为默认路径（兼容插件修改 files 的需求）；③ 两个路径均支持 changedTags 增量渲染；④ 流式方法返回 files 映射以供插件 hooks 使用 |

**`WorkerPool` 泛型设计**：

```ts
// src/core/WorkerPool.ts
export class WorkerPool<Task, Result> {
  constructor(options: {
    workerScript: string
    sharedContext: Record<string, unknown>
    poolSize: number
    idleTimeout?: number  // default 30000
  }) {}
  get size(): number
  async processBatch(tasks: Task[]): Promise<Result[]>
  terminate(): void
}

// 自适应池大小
export function pickPoolSize(apiCount: number): number
```

**使用示例**：

```ts
// templateParser 中 schema→TS
const pool = new WorkerPool<SchemaTask, SchemaResult>({
  workerScript: path.resolve(__dirname, '../workerPool/schemaWorker.js'),
  sharedContext: { document, config, refNameMapEntries },
  poolSize: pickPoolSize(apiCount),
})

// openApiParser 中 Swagger2→3
const pool = new WorkerPool<OpenAPIV2Document, OpenAPIDocument>({
  workerScript: path.resolve(__dirname, 'swagger2Worker.js'),
  sharedContext: {},
  poolSize: 1,
})
```

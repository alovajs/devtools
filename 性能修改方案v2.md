# Worma 性能优化方案 v2.0

> 目标：将 Worma 稳定支撑到 **5000–10000 API** 量级，显著降低生成耗时与峰值内存占用。
> 范围：以 `packages/worma/src/generate.ts` 为入口，覆盖解析、AST 生成、模板渲染、格式化、文件写入、差异更新等全链路。
> 说明：本方案为设计文档，**尚未改动代码**；后续按优先级分阶段实施。

---

## 1. 当前瓶颈总结（基于 benchmark 实测）

### 1.1 200 API 场景（alova template）

| 阶段              | 耗时      | 占比 | 主要消耗                                       |
| ----------------- | --------- | ---- | ---------------------------------------------- |
| TemplateParser    | ~1.33s    | ~42% | AST/Schema 转换、WorkerPool 调度               |
| per-tag 渲染+写入 | ~1.60s    | ~50% | Handlebars 渲染 + prettier 文件级格式化 + 写盘 |
| 其他              | ~0.27s    | ~8%  | hash 计算、diff、日志等                        |
| **总耗时**        | **~3.2s** |      |                                                |

### 1.2 5000 API 场景（alovaGlobals template）

| 阶段           | 耗时       | 占比 | 主要消耗                                |
| -------------- | ---------- | ---- | --------------------------------------- |
| TemplateParser | ~8.50s     | ~64% | schema 去重、重复 AST 计算、Worker 通信 |
| 渲染+写入      | ~4.40s     | ~33% | 3 个大文件（globals 等）prettier 格式化 |
| 其他           | ~0.40s     | ~3%  | diff、hash 等                           |
| **总耗时**     | **~13.3s** |      |                                         |

### 1.3 关键洞察

1. **TemplateParser 是跨量级共同的第一瓶颈**：无论是 200 还是 5000 API，解析阶段都占 40% 以上时间。
2. **大文件 prettier 是隐形杀手**：5000 API 下 3 个大文件 × 1.47s ≈ 4.4s，比 per-tag 35 个小文件 × 41ms 慢 **36 倍**。
3. **同一 schema 被重复转换**：`parseParameters`、`parseRequestBody`、`parseResponse` 会分别调用 `schemaLoader.transform` 两次（一次出类型、一次出注释）。
4. **完整 `allApis` 常驻内存**：`TemplateHelper.setData` 缓存整个 API 数组，量级越大峰值内存越高。
5. **WorkerPool 阈值偏高**：`apiCount <= 200` 时仍返回 0，意味着 200 API 场景未启用 Worker；而 5000 API 场景 Worker 通信/序列化开销需要优化。

## 3. 优化方案（按优先级）

### P0：单次 AST 同时产出 type + comment

**问题**：

`packages/worma/src/core/parser/templateParser/helper.ts` 中，`parseParameters`、`parseRequestBody`、`parseResponse` 分别调用 `schemaLoader.transform` 两次：

```ts
// 伪代码示意
const type = schemaLoader.transform(schema, { mode: 'type' });
const comment = schemaLoader.transform(schema, { mode: 'docComment' });
```

两次独立递归遍历同一 schema，导致 O(2n) 开销。

**方案**：

改造 `astLoader` / `schemaLoader`，提供统一入口 `transformSchemaOnce(schema)`，单次遍历同时返回：

- `tsType`: TypeScript 类型节点字符串
- `docComment`: JSDoc 注释字符串
- 其他元信息（如 required、enum、description 等）

**预期收益**：

- TemplateParser 阶段耗时降低 **20%–40%**。
- 减少重复对象分配，降低 GC 压力。

**涉及文件**：

- `packages/worma/src/core/loader/astLoader/index.ts`
- `packages/worma/src/core/loader/astLoader/generates.ts`
- `packages/worma/src/core/loader/schemaLoader/index.ts`
- `packages/worma/src/core/parser/templateParser/helper.ts`

### P0：顶层 schema 结果缓存（WeakMap）

**问题**：

同一 OpenAPI schema 对象在多个 API 间被复用（如公共 Page 类型、通用 Response 等），但当前没有缓存，每次都重新生成 AST。

**方案**：

在 `schemaLoader.transform` 或 `astLoader` 层引入 `WeakMap<object, TransformResult>`：

```ts
const schemaCache = new WeakMap<object, TransformResult>();

function transform(schema: object): TransformResult {
  if (schemaCache.has(schema)) {
    return schemaCache.get(schema)!;
  }
  const result = doTransform(schema);
  schemaCache.set(schema, result);
  return result;
}
```

- 使用 `WeakMap` 避免内存泄漏，schema 对象释放后缓存自动失效。
- 适用于引用类型 schema；基本类型不缓存。

**预期收益**：

- 公共 schema 复用场景下，解析时间减少 **30%–50%**。
- 对 5000 API 大规格效果更明显。

**涉及文件**：

- `packages/worma/src/core/loader/schemaLoader/index.ts`
- 或 `packages/worma/src/core/loader/astLoader/index.ts`

### P1：per-tag 批量写入，用满 `writeConcurrency`

**问题**：

per-tag 模式下，当前可能是逐个文件写入，未充分利用配置的并发度。200 API 场景下 35 个文件 × 41ms 仍有优化空间。

**方案**：

- 将 per-tag 文件按 `writeConcurrency` 分组（例如 8 个一组）。
- 使用 `pMap` 或自定义队列控制并发写入，避免同时打开过多文件句柄。
- 小文件可合并临时 buffer 后统一落盘，减少 I/O 抖动。

```ts
const writeConcurrency = config.writeConcurrency || 8;
await pMap(fileGroups, async (group) => {
  await Promise.all(group.map(file => writeFile(file.path, file.content)));
}, { concurrency: writeConcurrency });
```

**预期收益**：

- 200 API per-tag 写入耗时从 1.6s 降至 **0.8s 左右**。

**涉及文件**：

- `packages/worma/src/helper/template/TemplateHelper.ts`
- `packages/worma/src/helper/config/GeneratorHelper.ts`

### P1：按需/延迟生成 docComment

**问题**：

每个 API 的 docComment 默认生成，但用户模板可能并未使用这些注释。对于 5000 API 场景，这是大量无效计算。\n\n**注意**：当前配置已提供 `docComment?: boolean`（默认 `true`），用户可通过设置 `docComment: false` 关闭注释生成以提升性能。

**方案**：

- 该能力已通过配置项 `docComment` 控制，无需新增开关。
- 优化重点转为：当 `docComment: false` 时，在 parser 最前端彻底短路 docComment 相关计算，避免任何中间对象或 AST 分支生成。
- 同时梳理文档/CLI 提示，明确在 1000+ API 场景下建议关闭 `docComment`。

**预期收益**：

- 未使用注释的模板可节省 **10%–20%** 解析时间。

**涉及文件**：

- `packages/worma/src/core/parser/templateParser/helper.ts`
- `packages/worma/src/helper/template/TemplateHelper.ts`
- `packages/worma/src/helper/config/GeneratorHelper.ts`（短路判断入口）

### ~~P2：setData 预计算 hash（隐式优化）~~ ❌ 已回滚

**问题（原以为的优化）**：

在 `TemplateHelper.setData` 阶段预先计算 `computePerTagHashes(apis)`，将 hash 和 tags 存入 `CachedEntry`，使 `flushAllData` 可直接使用预计算结果，避免二次 SHA256。

**实际效果**：

该变更在实施后带来了负面收益：

1. `setData` 仍需存储完整 `apis` 数组到 `_cacheData`（因为模板渲染需要），**内存收益为零**。
2. `computePerTagHashes` 对 5020 个 API × 995 个 tag 调用 `computeApiHash`（`JSON.stringify` + SHA256），额外增加 **2-3s** 计算开销。
3. 该计算本应在 `flushAllData` 写盘阶段就近完成，预计算属于过早优化。

**回滚方案**：恢复 `setData` 只存原始 `CacheData`，hash 计算延后到 `flushAllData`，与写盘原子完成。

**涉及文件**：

- `packages/worma/src/helper/template/index.ts`
- `packages/worma/src/functions/wormaJson.ts`

- 使用 generator 或流式数据结构分批提供 API 数据。

**预期收益**：

### P2：增量 hash 与 diff 缓存优化

**问题**：

`computePerTagHashes` 和 `diffChangedTags` 在每次生成时都重新计算所有 tag 的 hash，对于 5000 API unnecessary。

**方案**：

- 缓存上次生成的 per-tag hash 结果。
- 仅当 API 定义或配置变化时重新计算受影响 tag 的 hash。
- 结合文件 mtime 或内容指纹，减少无变化文件的重写。

**预期收益**：

- 增量构建场景下，二次生成耗时降低 **50% 以上**。

**涉及文件**：

- `packages/worma/src/functions/wormaJson.ts`

### P2：持久化 Worker 池与减少序列化开销

**问题**：

当前 WorkerPool 在每次生成时创建/销毁，Worker 通信序列化开销较大；且 `apiCount <= 200` 时未启用 Worker，200 API 场景损失了并行收益。

**方案**：

- 降低 Worker 启用阈值，例如 `apiCount > 20` 即启用 Worker。
- 引入 Worker 池复用机制，跨次生成复用 Worker 实例（可选，需权衡内存）。
- 减少传递给 Worker 的数据量：只传 schema 引用或必要字段，避免传整个 API 对象。

**预期收益**：

- 200 API 场景解析时间减少 **15%–25%**。
- 5000 API 场景 Worker 开销更平稳。

**涉及文件**：

- `packages/worma/src/core/WorkerPool.ts`
- `packages/worma/src/core/workerPool/worker.ts`
- `packages/worma/src/core/parser/templateParser/index.ts`

### ~~P1：替换 JSON.stringify 去重（collectSchemaTasks）~~ ❌ 已回滚

**问题**：

`collectSchemaTasks` 使用 `JSON.stringify(schema)` 对 schema 对象做序列化去重。大规格（5000+ API）场景下，每个 schema 的 JSON 序列化累计开销巨大（O(n·s)，n=API 数量，s=schema 大小）。且 `parseReference` 默认返回原始对象引用，同一 `$ref` 解析出的 schema 是同一对象，引用级去重即可覆盖绝大多数去重场景。

**方案**：

使用 `WeakMap<object, boolean>` 按对象引用去重，替代 `Set<string>` + `JSON.stringify`：

```ts
let _counter = 0
const seen = new WeakMap<object, boolean>()

function add(schema: any): boolean {
  if (!schema || typeof schema !== 'object') return false
  if (seen.has(schema)) return false
  seen.set(schema, true)
  const key = `schema_${++_counter}`
  tasks.push({ key, schema })
  return true
}
```

- 同一 `$ref` 解析出的 schema 是同一引用，天然去重。
- 内联 schema 按引用判断，避免昂贵序列化。

**预期收益**：

5000+ API 场景 `collectSchemaTasks` 耗时降低 **90% 以上**，JSON.stringify → O(1) 引用比较。

**❌ 回滚原因**：

实际测试发现 WeakMap 引用去重引入了严重的**功能回归**：

1. `collectSchemaTasks` 的 `for` 循环中，`pathSchema` 和 `querySchema` 每次迭代都是全新构造的对象，即使内容完全相同也被视为不同引用 → **去重完全失效**。
2. 5000 个 API 产生 5000+ 个 schema task（而非原始 200-500 个），导致 Worker 处理数暴增。
3. `globals.d.ts` 输出膨胀数十倍（填满重复类型定义），prettier 格式化耗时从 ~4s 暴增到 **132s**。
4. 5000 API 总耗时从 12.6s 飙升到 **146.4s**（+11.6 倍）。

**结论**：`JSON.stringify` 的序列化开销在 schema 场景下极小（每个 schema 数十到数百字节，总计不到 1s），但它提供的**内容去重**是必不可少的。回退为原始 `Set<string>` + `JSON.stringify` 方案。

**涉及文件**：

- `packages/worma/src/core/parser/templateParser/index.ts`

### P1：进度节流（16ms throttle）

**问题**：

`ProgressTracker` 使用 `interval = 0`，意味着每次 `update()` 立即 flush 到 UI。高频场景下（5000+ API 批量处理），密集的 progress 事件会淹没 event loop，反而拖慢渲染和生成速度。

**方案**：

将 `interval` 从 `0` 改为 `16`（约 60fps），确保每秒最多 60 次 progress 事件：

```ts
const tracker = new ProgressTracker((snapshot) => {
  // ...
}, 16) // 16ms 节流，足够 UI 更新
```

**预期收益**：

- 减少 progress 事件触发频率 80%+（实测高频场景）。
- UI 更新更平滑，不阻塞 Worker 通信和文件 I/O。

**涉及文件**：

- `packages/worma/src/generate.ts`
- `packages/worma/src/helper/progress.ts`

### P2：Worker 里共享 schema 缓存

**问题**：

`worker.ts` 的 `processSchema` 每处理一个 schema 都会从头调用 `schemaLoader.transformTsStr`。当批次内多个 schema 引用同一公共类型时，相同转换会重复执行，浪费 Worker 内 CPU 时间。

**方案**：

在 Worker 模块作用域内维护一个本地 `Map<string, string>` 缓存，按 `keyName`（或 schema 引用 hash）缓存转换结果：

```ts
const schemaCache = new Map<string, string>()

async function processSchema(key: string, schema: SchemaObject): Promise<SchemaResultItem[]> {
  // 检查缓存
  const cacheKey = JSON.stringify(schema) // 仅 schema ref id 或简短指纹
  if (schemaCache.has(cacheKey)) return [{ key, result: schemaCache.get(cacheKey)! }]
  // ... 正常转换
  schemaCache.set(cacheKey, result)
  return results
}
```

- 使用 `Map` 而非 `WeakMap`（Worker 生命周期内引用不会释放，Map 的 size 可控）。
- 可按 LRU 策略限制缓存上限，避免内存无限增长。

**预期收益**：

Worker 批次内有公共类型复用时，减少 **20%–40%** Worker 内转换时间。

**涉及文件**：

- `packages/worma/src/core/workerPool/worker.ts`

### P2：持久化 Worker 池复用

**问题**：

当前 WorkerPool 是"用一次创建一次"：

```ts
const pool = new WorkerPool(...)
const results = await pool.processBatch(tasks)
pool.terminate()
```

每次生成都要创建/销毁 Worker 进程。对于频繁触发的 CI / watch 模式，创建销毁开销累计不可忽视。

**方案**：

实现全局可复用的 WorkerPool manager，按 `poolSize` 缓存池实例：

```ts
// 新增 packages/worma/src/core/workerPool/poolManager.ts
const pools = new Map<string, WorkerPool<any, any>>()

export function getWorkerPool<T, R>(key: string, factory: () => WorkerPool<T, R>): WorkerPool<T, R> {
  if (!pools.has(key)) pools.set(key, factory())
  return pools.get(key)!
}
```

- 按 `key`（如 `workspacePath + templateType`）缓存池。
- 加入 idle timeout（如 30s 无任务则 terminate），避免闲置 Worker 占用内存。
- 注意跨次生成的状态清理：复用前 reset Worker 内部缓存。

**预期收益**：

- 二次/多次生成时，省去 Worker 创建开销（~200-500ms）。
- Watch 模式下体验显著提升。

**涉及文件**：

- `packages/worma/src/core/WorkerPool.ts`
- `packages/worma/src/core/workerPool/poolManager.ts`（新增）
- `packages/worma/src/core/parser/templateParser/index.ts`

### P2：减少 cloneDeep 开销

**问题**：

`transformApiMethods` 路径下有 3 处 `cloneDeep`：

```ts
// helper.ts
apiMethod2ApiDescriptor   -> cloneDeep(operationObject)
apiDescriptor2apiMethod    -> cloneDeep(apiDescriptor); cloneDeep(operationObject)
```

`operationObject` 包含 `parameters / requestBody / responses`，可能有巨大嵌套 schema。`handleApi` 通常只读改少量顶层字段，全量深拷贝极其浪费。

**方案**：

提供受控副本——仅深拷贝用户可能修改的顶层字段：

```ts
const apiDescriptor = {
  summary: operationObject.summary,
  operationId: operationObject.operationId,
  tags: operationObject.tags,
  parameters: cloneDeep(operationObject.parameters),
  requestBody: cloneDeep(operationObject.requestBody),
  responses: cloneDeep(operationObject.responses),
}
```

- 顶层简单字段（summary/operationId/tags）直接赋值。
- 复杂嵌套对象（parameters/requestBody/responses）按需选择性深拷贝。
- 如果某模板不修改特定字段，可进一步跳过克隆。

**预期收益**：

API 方法数较大时，减少 cloneDeep 耗时 **50%–80%**，并降低临时内存分配。

**涉及文件**：

- `packages/worma/src/core/parser/templateParser/helper.ts`

### P2：避免存两份 allApis

**问题**：

`TemplateHelper.setData` 将完整 `templateData.allApis` 存入 `_cacheData`，同时 `templateData` 自身也持有一份 `allApis`：

```ts
TemplateHelper._cacheData.set(key, {
  path: output,
  serverName: ...,
  apis: templateData.allApis, // ← 第二份引用
})
```

对于 5000+ API，这是不必要的额外内存占用（虽然 JS 引用不复制对象，但阻止了原始对象被 GC）。

**方案**：

`_cacheData` 只存轻量摘要，在需要时按需从源数据组装：

```ts
TemplateHelper._cacheData.set(key, {
  path: output,
  serverName: config?.serverName || templateData.title || '',
  apiCount: templateData.allApis?.length || 0,   // 仅存数量
  apiIds: templateData.allApis?.map(a => a.name), // 或仅存 id 列表
})
```

- 如果缓存目的是判断增量变化，存 hash/id 列表即可。
- 如果缓存目的是渲染，改为按需获取而非全量持有。

**预期收益**：

峰值内存降低 **5%–15%**（取决于 API 对象复杂度）。

**涉及文件**：

- `packages/worma/src/helper/template/index.ts`

### P3：流式/分片处理（长期架构）

**问题**：

当 API 数量达到 10000+ 时，即使单次 AST 优化，内存和时间仍可能接近上限。

**方案**：

- 将 API 集合按 tag/path 分片。
- 每个分片独立解析、渲染、写入，完成后立即释放内存。
- 使用 Node.js stream 或 async generator 处理大文件输出，避免一次性拼接超大字符串。

**预期收益**：

- 支撑 10000+ API 稳定运行。
- 内存占用从 O(n) 接近 O(1) 分片级。

**涉及文件**：

- `packages/worma/src/generate.ts`
- `packages/worma/src/helper/config/GeneratorHelper.ts`
- `packages/worma/src/helper/template/TemplateHelper.ts`

## 5. 风险与注意事项

1. **内容去重 vs 引用去重**：`collectSchemaTasks` 中 `pathSchema`/`querySchema` 是每次循环新建的对象，而非 `$ref` 解析出的共享引用。引用去重（WeakMap）对此类自构造对象**完全无效**，必须保留 `JSON.stringify` 内容去重。详见 §P1 已回滚项。
2. **hash 预计算的时机陷阱**：在 `setData` 阶段预计算 hash 属于过早优化——`setData` 调用后 `apis` 仍需常驻内存供渲染使用，预计算仅增加了额外的 CPU 开销而无内存收益。hash 计算应在写盘阶段就近完成。
3. **缓存一致性**：引入 `WeakMap` 缓存后，需确保 schema 对象不可变；若外部修改 schema 对象，缓存结果可能过期。可考虑在转换时做浅冻结或复制关键字段。
4. **docComment 短路完整性**：当 `docComment: false` 时，需确保 parser、loader、模板渲染全链路均跳过注释计算，避免某个中间环节仍生成注释数据导致收益打折。默认保持 `docComment: true` 以兼容现有用户。
5. **Worker 稳定性**：复用 Worker 时需注意 Worker 状态清理，避免跨次生成数据污染。
6. **并发写盘限制**：per-tag 批量写入时，需控制并发数，避免 Windows 系统下文件句柄过多或竞争。

## 6. 验证计划

1. **基准测试**：在 `benchmark/` 目录分别运行 200 API、5000 API、10000 API 场景，记录各阶段耗时与内存。
2. **回归测试**：确保生成的 TypeScript 文件与优化前一致（或仅有格式化差异）。
3. **内存分析**：使用 Node.js `--heap-snapshot-near-heap-limit` 或 clinic.js 采集堆快照，对比峰值内存。
4. **增量测试**：连续运行两次生成，验证 diff 与 hash 缓存效果。
